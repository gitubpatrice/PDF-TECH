# Audit complet — PDF Tech v1.8.0

**Date** : 2026-05-03
**Périmètre** : application Flutter Android, 23 outils PDF, 100% offline (sauf Google Drive opt-in)
**Sources auditées** : `pubspec.yaml`, `android/app/build.gradle.kts`, `AndroidManifest.xml`, `res/xml/*`, `MainActivity.kt`, `proguard-rules.pro`, intégralité de `lib/`
**Référentiels** : OWASP Mobile Top 10 2024, MASVS-L1

---

## Synthèse

L'app présente une posture sécurité globalement **bonne pour une app PDF Android** : manifest durci, FileProvider correctement scopé, validation des entrées PDF (magic bytes + size cap), isolement des opérations lourdes via `Isolate.run` dans `PdfToolsService`, MethodChannel `sendToPackage` protégé par `isAllowedPath` (canonicalisation + comparaison root). Les principales faiblesses portent sur :

1. La permission `MANAGE_EXTERNAL_STORAGE` (sur-permission Play Store-bannie).
2. Plusieurs **écrans d'outils contournant `PdfToolsService`** : ils parsent le PDF directement sur le main isolate, sans `_safeReadPdf` (pas de cap de taille, pas de vérif magic bytes).
3. Le **scan récursif global** (`_walk`) sur le main isolate, qui peut bloquer l'UI sur stockage rempli.
4. Couverture de tests quasi nulle (1 smoke test).

Aucun secret committé, pas de WebView, pas de cleartext, pas de debuggable, pas de keystore dans le repo.

## 🟠 Élevé

### F1. Sur-permission `MANAGE_EXTERNAL_STORAGE`

**Fichier** : `android/app/src/main/AndroidManifest.xml:10`

```xml
<uses-permission android:name="android.permission.MANAGE_EXTERNAL_STORAGE"
    tools:ignore="ScopedStorage" />
```

**Impact** :
- Permission "All files access" : Google Play interdit cette permission sauf justification stricte (file manager, antivirus, sauvegarde) avec déclaration formulaire. Pour une app PDF, le rejet est probable.
- En cas de compromission, l'app a accès à *tous* les fichiers utilisateurs (photos, autres docs sensibles), bien au-delà de son besoin (lire/écrire des PDF dans Documents/Download).
- Surface d'attaque amplifiée : un PDF malveillant + une vulnérabilité Syncfusion → exfil de tout le stockage.

**Correctif** :
1. **Idéal** : remplacer par SAF (`ACTION_OPEN_DOCUMENT` / `ACTION_OPEN_DOCUMENT_TREE`) + persistable URI permission. Le module `files_tech_core` semble déjà utiliser RecentFiles ; passer aux `Uri` plutôt qu'aux `String path`.
2. **Réaliste à court terme** : justifier formellement Play (catégorie "File Manager") et limiter strictement les usages au scan récursif. Documenter dans `PRIVACY.md` que la perm n'est demandée qu'au tap utilisateur sur "Trouver mes PDFs".
3. **Atténuation** : l'app demande déjà la perm via dialog explicatif (`main.dart:202-227`) — bien. Mais le scan récursif `_walk` (`home_screen.dart:537`) ne devrait scanner que `Download/`, `Documents/`, dossiers déclarés, pas toute la racine `/storage/emulated/0`.

### F2. 14/21 écrans d'outils contournent `_safeReadPdf`

**Fichiers concernés** (toutes les lignes appellent `File(...).readAsBytes()` puis `PdfDocument(inputBytes: ...)` directement) :

| Fichier | Ligne |
|---|---|
| `lib/screens/tools/compare_screen.dart` | 41-42 |
| `lib/screens/tools/header_footer_screen.dart` | 38-39, 60-61 |
| `lib/screens/tools/stamp_screen.dart` | 65-66 |
| `lib/screens/tools/signature_screen.dart` | 49-50 |
| `lib/screens/tools/form_fill_screen.dart` | 51-52, 362 |
| `lib/screens/tools/reorder_pages_screen.dart` | 33-34, 91-92 |
| `lib/screens/tools/page_numbers_screen.dart` | 31-32, 61-62 |
| `lib/screens/tools/extract_images_screen.dart` | 38 |
| `lib/screens/tools/ocr_screen.dart` | 56-57 |
| `lib/screens/tools/export_images_screen.dart` | 34-35 |
| `lib/screens/tools/metadata_screen.dart` | 38-39, 56-57 |
| `lib/screens/tools/delete_pages_screen.dart` | 26-27, 49-50 |
| `lib/screens/tools/pdf_annotate_screen.dart` | 191-192 |
| `lib/screens/tools/create_pdf_screen.dart` | 202 (image bytes seulement) |

**Impact** :
- Pas de cap 200 Mo → un PDF de 2 Go ouvert depuis SAF charge tout en RAM : **OOM kill** garanti sur appareils 3-4 Go RAM.
- Pas de validation magic bytes → un fichier `.pdf` truqué (renommé) sera passé à Syncfusion. Si Syncfusion a une CVE de parsing, l'app crash voire exfile via JS embedded action (Syncfusion gère AcroForm/JS — vérifier qu'il est désactivé par défaut).
- Pas de `Isolate.run` → toute la durée du parse fige l'UI, et si le PDF est une "PDF bomb" (objets cycliques, milliers de pages vides), l'app ne répond plus → ANR.

**Correctif** :
Centraliser systématiquement via `PdfToolsService._safeReadPdf` rendu **public** (renommer en `safeReadPdf`) :

```dart
// pdf_tools_service.dart
static Future<Uint8List> safeReadPdf(String path) async { /* idem */ }
```

Puis dans chaque écran :
```dart
final bytes = await PdfToolsService.safeReadPdf(_path!);
final result = await Isolate.run(() => _myIsolate(bytes, ...));
```

Un refactor de cette ampleur peut être grouped : créer un helper `pdf_isolate_runner.dart` qui prend `(path, isolateFn)` et factorise lecture sécurisée + isolate + écriture.

### F3. Parsing PDF sur main isolate dans la majorité des écrans `tools/*`

Identique à F2 mais centré performance/UX. `PdfToolsService` fait correctement `Isolate.run` pour merge/split/protect/rotate/watermark/compress/decrypt/getPageCount, mais les écrans suivants ne l'utilisent pas et parsent en main :

`compare_screen`, `delete_pages_screen`, `reorder_pages_screen`, `page_numbers_screen`, `metadata_screen`, `header_footer_screen`, `stamp_screen`, `signature_screen`, `form_fill_screen`, `extract_images_screen`, `pdf_annotate_screen`, `images_to_pdf_screen`, `create_pdf_screen`, `ocr_screen` (parse+extract), `export_images_screen`.

**Impact** : freeze UI 5-30s sur PDF de 50-200 Mo, jankiness sur petits fichiers, ANR risk.

**Correctif** : étendre le pattern `Isolate.run(() => _xxxIsolate(...))` de `PdfToolsService` à tous les outils. Pour chaque outil, déplacer la logique métier en méthode statique top-level ou statique de la classe service, et passer `Uint8List` (sendable). Les opérations `pdfx` (rendu page) ne sont **pas** trivialement isolatables (FFI handles natifs) ; envisager `compute()` ou keep en main mais throttler.

## 🟡 Moyen

### F4. OCR / ExportImages : rendu pages + ML Kit sur main isolate

**Fichiers** : `lib/screens/tools/ocr_screen.dart:97-127`, `lib/screens/tools/export_images_screen.dart:66-96`

`pdfx.PdfDocument.openFile()` + `page.render()` + `_rawToPng()` + `recognizer.processImage()` enchaînés sans yield → boucle qui fige l'UI sur PDF 50+ pages. La progress bar `_processedPages` ne s'anime quasi pas car `setState` est noyé.

**Correctif** :
- Ajouter `await Future.delayed(Duration.zero)` ou mieux `await SchedulerBinding.instance.endOfFrame` entre chaque page pour rendre la main.
- Mieux : exécuter `_rawToPng` (CPU pur via `dart:ui`) dans un isolate.
- Pour ML Kit, c'est natif — déjà sur thread natif côté Android. OK.
- Considérer un cap de pages (par ex. 500) pour éviter l'OOM lors du rendering 4x.

### F5. `_extractJpegs` : scan O(n) octet-par-octet sur main isolate, sans cap

**Fichier** : `lib/screens/tools/extract_images_screen.dart:68-91`

```dart
List<Uint8List> _extractJpegs(Uint8List data) {
  final results = <Uint8List>[];
  int i = 0;
  while (i < data.length - 3) { ... data.sublist(start, j+2) ... }
}
```

**Impact** :
- Sur un PDF de 100 Mo, `data.length = 100_000_000`. La double boucle `while/while` est O(n²) dans le pire cas (faux positif `FF D8 FF` sans `FF D9` ensuite → relance la boucle interne sur tout le reste). Freeze multi-seconde.
- `data.sublist()` *copie* — multiplication de la mémoire si beaucoup d'images.
- Pas de garde de taille en entrée (pas de `_safeReadPdf`).

**Correctif** :
1. Wrapper l'appel dans `Isolate.run(() => _extractJpegs(bytes))`.
2. Sur faux positif, faire `i = j` (pas `i++` mais sauter au max sondé) pour éviter O(n²). Ou utiliser `_safeReadPdf`.
3. Cap : si `results.length > 1000` → break + warn user.

### F6. Google Drive download : `bytes.addAll(chunk)` en RAM

**Fichier** : `lib/services/google_drive_service.dart:65-78`

```dart
final bytes = <int>[];
await for (final chunk in media.stream) {
  bytes.addAll(chunk);
}
...
await File(localPath).writeAsBytes(Uint8List.fromList(bytes));
```

**Impact** : un PDF Drive de 500 Mo charge intégralement en RAM, plus une copie via `Uint8List.fromList`. OOM probable.

**Correctif** : streamer directement vers le fichier :
```dart
final sink = File(localPath).openWrite();
try {
  await media.stream.pipe(sink);
} finally {
  await sink.close();
}
```

### F7. `compressPdf` / `getPageCount` : double allocation inutile

**Fichier** : `lib/services/pdf_tools_service.dart:88, 118, 141, 162, 212, 238, 300, 321`

```dart
final out = Uint8List.fromList(merged.saveSync());
```

`saveSync()` retourne déjà `List<int>` ; `Uint8List.fromList` re-copie. Sur un PDF de 100 Mo, c'est 100 Mo de copie RAM superflue dans l'isolate.

**Correctif** :
```dart
final saved = merged.saveSync();
final out = saved is Uint8List ? saved : Uint8List.fromList(saved);
```
Ou directement passer `List<int>` via `await File(path).writeAsBytes(saved)` (qui accepte `List<int>`).

### F8. `decrypt_screen.dart` : double `setState` sans re-check `mounted`

**Fichier** : `lib/screens/tools/decrypt_screen.dart:74-86`

```dart
setState(() => _isProcessing = false);
messenger.showSnackBar(SnackBar(...));
setState(() {  // ← second setState après gap async potentiel
  _path = null; _name = null; _passwordCtrl.clear();
});
```

Entre les deux `setState` il n'y a pas d'await, OK ici. Mais `_passwordCtrl.clear()` à l'intérieur d'un `setState` est inutile (le controller n'est pas state Flutter).

**Correctif** : mineur, fusionner les `setState` :
```dart
setState(() {
  _isProcessing = false;
  _path = null; _name = null;
});
_passwordCtrl.clear();
```

### F11. `_AllPdfsScreen` : `statSync()` à chaque comparaison du tri

**Fichier** : `lib/screens/home_screen.dart:522-523`

```dart
found.sort((a, b) => b.statSync().modified.compareTo(a.statSync().modified));
```

`sort()` appelle le comparator O(n log n) fois ; chaque appel fait 2 `statSync()` (I/O sync). Sur 1000 PDFs, c'est ~20000 `statSync` synchrones sur main isolate.

**Correctif** : pré-calculer :
```dart
final withMtime = found.map((f) => (f, f.statSync().modified)).toList()
  ..sort((a, b) => b.$2.compareTo(a.$2));
final sorted = withMtime.map((e) => e.$1).toList();
```

Et plus loin (`home_screen.dart:1269`, `pdf_folder_screen.dart:62, 173`) : même problème sur le `ListView.builder` qui appelle `f.statSync()` dans `itemBuilder` — appel I/O à chaque scroll. Cacher les `FileStat` dans une struct `(File, FileStat)` lors du scan.

**Fichier** : `lib/screens/tools/pdf_annotate_screen.dart:190-209`

```dart
Future<String> _flatten() async {
  final bytes = await File(widget.path).readAsBytes();
  final doc = PdfDocument(inputBytes: bytes);
```

## 🟢 Bonnes pratiques observées

### B1. Hardening manifest
- `allowBackup="false"` + `fullBackupContent="false"` + `data_extraction_rules.xml` qui exclut tous les domaines pour Android 12+.
- `usesCleartextTraffic="false"` + `network_security_config` strict (system trust anchors uniquement, domaines whitelistés `api.github.com` / `googleapis.com`).
- `taskAffinity=""` sur MainActivity → mitige task hijacking.
- `MainActivity` exported uniquement pour `MAIN/LAUNCHER` + `VIEW application/pdf`. FileProvider `exported="false"` avec `grantUriPermissions="true"` — correct.

### B2. `_safeReadPdf`
**Fichier** : `lib/services/pdf_tools_service.dart:24-45`. Cap 200 Mo + magic bytes `%PDF-` + size minimale + exception typée `PdfValidationException`. Très bien — à étendre à tous les écrans (cf. F2).

### B3. `MainActivity.isAllowedPath`
**Fichier** : `android/app/src/main/kotlin/com/pdftech/pdf_tech/MainActivity.kt:29-39`. Canonicalisation (`canonicalFile` suit symlinks), comparaison par `startsWith(root + separator)` (évite `/storage/emulated/0bad` qui matcherait `/storage/emulated/0`). Excellent contre les attaques où Dart enverrait un path forgé.

### B4. Secrets non committés
`.gitignore` couvre `/android/key.properties` et `/android/app/keystore.jks`. Build script avec fallback env vars (`PDFTECH_*`) propre.

### B5. ProGuard/R8 release
`isMinifyEnabled = true` + `isShrinkResources = true` + règles ciblées pour Syncfusion/MLKit/pdfx (qui font du reflection). Pas de `-keep class **` global.

### B6. Crypto solide
`PdfEncryptionAlgorithm.aesx256Bit` pour `protectPdf`, owner password `Random.secure()` 32 chars dans alphabet de 60+. Conforme.

### B7. Sanitisation Drive
`google_drive_service.dart:73-76` : strip `[\\/\x00-\x1f]` + leading dots → anti path traversal lors du download. Très bien.

### B8. OCR cleanup garanti
`ocr_screen.dart:97-136` : sous-dossier dédié `ocr_session_<ts>` purgé en `finally` même sur exception. Évite la fuite de PNG décompressés (= contenu PDF reconstituable) dans le cache app.

### B9. Update info SHA-256
`home_screen.dart:82-113` affiche le SHA-256 attendu de l'APK pour vérification utilisateur post-download. Bonne hygiène supply-chain pour distribution APK side-load.

### B10. Avertissements crypto explicites
`decrypt_screen.dart:42-65` : dialog modal "Le PDF déchiffré sera en clair" avant l'opération. UX sécurité sérieuse.

### B11. PopScope avec save annotations
`pdf_viewer_screen.dart:206-214` : `PopScope` + dialog "Annotations non sauvegardées" évite la perte de données.

## Conclusion

PDF Tech v1.8.0 est une app **soigneusement durcie au niveau Android/manifest/keystore**, avec un service central (`PdfToolsService`) qui montre le bon pattern (validation + isolate). Le défaut majeur est que **ce pattern n'a pas été propagé aux 14 autres écrans d'outils**, qui restent vulnérables à OOM/ANR sur PDF volumineux ou malveillants. La correction est mécanique (refactor) et ne demande pas de redesign.

La permission `MANAGE_EXTERNAL_STORAGE` reste le seul vrai point bloquant pour Play Store ; en mode side-load (distribution APK GitHub Releases), c'est acceptable mais à documenter.

Aucune faille critique exploitable à distance. Aucun secret exposé. Crypto correcte. FileProvider scopé. Code globalement propre, sans `WebView`, sans `JavascriptInterface`, sans deep link à risque.
